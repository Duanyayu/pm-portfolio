# A3：PRD → WBS → Epic → Story 拆解

> 将 [A2 PRD](./A2-prd-from-test-report.md) 中的 4 个 Epic 按 PM Skills `epic-breakdown-advisor` 的 Humanizing Work 9 拆分模式拆解为可执行的 Story。
> 参考：本项目 `docs/jira/wbs-to-jira-mapping.md` 的 Jira 映射规范。

---

## WBS（Work Breakdown Structure）总览

```
测试环境兼容性优化专项
├── E1: 测试设备兼容性保障（【短期】规避风险）
│   ├── S1.1: TU36 型号测试流程迁移至 1 楼海外台体
│   ├── S1.2: 全型号功耗特征 × 设备带载能力匹配审查
│   └── S1.3: 测试环境异常处理 SOP 更新
│
├── E2: 电源柜设备替换与维修（【短期】消除瓶颈）
│   ├── S2.1: 电源柜 1 号停用 + 供应商维修联系
```

## Epic 1: 测试设备兼容性保障

**Epic Hypothesis:** 我们相信通过将 TU36 测试迁移至已验证兼容的台体并建立系统性的兼容性审查机制，可以消除当前 80% 测试场景的供电异常问题，使测试一次上电成功率从 ~5% 提升至 ≥95%。

**拆分模式：** Workflow Steps（工作流步骤）+ Business Rule Variations（不同表型 = 不同规则）

### Story 1.1: TU36 型号测试流程迁移至海外台体

| 字段 | 内容 |
|---|---|
| **User Story** | 作为测试工程师，我希望 TU36 型号的全性能测试能在 1 楼海外台体上稳定执行，以便我无需通过反复上电尝试来开始测试 |
| **Acceptance Criteria** | ① TU36 1号表和 2号表在海外台体上连续 3 次上电，三相电流均正常施加 ② 三相电流偏差 ≤ 0.5A ③ 全性能测试流程可完整执行，无中断 |
| **Priority** | P0 - 立即执行 |
| **Estimate** | 0.5 天 |
| **依赖** | 海外台体测试排期协调 |

### Story 1.2: 全型号功耗特征 × 设备带载能力匹配审查

| 字段 | 内容 |
|---|---|
| **User Story** | 作为测试部负责人，我希望获得一份"在测表型功耗 vs 可用设备带载能力"的完整匹配矩阵，以便我预判哪些型号组合可能存在类似 TU36 的兼容风险 |
| **Acceptance Criteria** | ① 列出所有在测表型的标称功耗（VA）② 列出所有可用电源柜的实测带载能力上限（A/相）③ 输出匹配矩阵，标注"安全"、"边界"、"不匹配"三档 ④ 对"边界"和"不匹配"项给出处理建议 |
| **Priority** | P1 - 本周内 |
| **Estimate** | 2 天 |
| **依赖** | 各表型功耗参数（部分需研发提供） |

### Story 1.3: 测试环境异常处理 SOP 更新

| 字段 | 内容 |
|---|---|
| **User Story** | 作为测试工程师，我希望在遇到类似"电源无法正常施加"的异常时，有标准化的排查流程——而非依赖个人经验反复尝试 |
| **Acceptance Criteria** | ① SOP 包含：问题分类（电压异常 / 电流异常 / 相位异常）、排查步骤（设备自检 → 更换参考表验证 → 更换设备验证 → 上报）、升级路径（什么情况下通知研发/采购）② SOP 经过至少一次 TU36 测试实战验证 |
| **Priority** | P1 |
| **Estimate** | 1 天 |
| **依赖** | Story 1.2 的匹配矩阵（作为 SOP 中判断"是设备问题还是产品问题"的参考依据） |

## Epic 2: 电源柜设备替换与维修

**Epic Hypothesis:** 我们相信通过替换或维修带载能力不足的电源柜并建立带载能力验收标准，可以从根本上消除"测试设备无法正常供电"这类问题的再次发生。

**拆分模式：** Major Effort（替换 1 台 → 推广至全部）

### Story 2.1: 电源柜 1 号停用 + 供应商维修联系

| 字段 | 内容 |
|---|---|
| **User Story** | 作为设备管理员，我需要将电源柜 1 号正式停用并联系供应商进行维修，以避免设备进一步损坏和数据失真 |
| **Acceptance Criteria** | ① 电源柜 1 号贴上"停用"标识并从测试区移除 ② 已联系供应商并提供故障现象和实验数据 ③ 供应商给出维修/替换方案和时间节点 |
| **Priority** | P0 |
| **Estimate** | 0.5 天 |
| **依赖** | 采购/设备管理部门配合 |

### Story 2.2: 替代电源柜选型评估

| 字段 | 内容 |
|---|---|
| **User Story** | 作为测试部/采购，我需要基于 TU36 事件的教训，选择一款带载能力有足够余量的替代电源柜 |
| **Acceptance Criteria** | ① 列出 ≥3 款候选设备 ② 每款标注标称带载能力、价格、供货周期 ③ 以 TU36 功耗（5.10VA）为基准，确认候选设备的单相带载能力 ≥ 1.5 倍安全系数 ④ 给出推荐选择及理由 |
| **Priority** | P1 |
| **Estimate** | 2 天 |
| **依赖** | 供应商报价和规格书获取 |

### Story 2.3: 新电源柜验收测试

| 字段 | 内容 |
|---|---|
| **User Story** | 作为测试工程师，我需要在新的电源柜投入使用前，验证其对所有在测表型（含 TU36 和 ZW22B）的供电能力是否达标 |
| **Acceptance Criteria** | ① 使用 TU36 1号表 + TU36 2号表 + 国网 ZW22B 三款参考表分别测试 ② 三相电流偏差 ≤ 0.5A ③ 连续 3 次上电均可正常施加 ④ 验收报告经测试部负责人签字 |
| **Priority** | P1 |
| **Estimate** | 1 天 |
| **依赖** | 新设备到货 |

## Epic 3: 设备选型标准建立

**Epic Hypothesis:** 我们相信通过建立"被测表型功耗 × 设备带载能力"匹配数据库和验收标准，可以防止未来再次发生因设备选型不当导致的测试环境兼容问题。

**拆分模式：** Simple/Complex（先做数据库和匹配矩阵的 MVP 版本，后续完善）

### Story 3.1: 被测表型功耗特征数据库

| 字段 | 内容 |
|---|---|
| **User Story** | 作为测试工程师，我希望在一个统一的地方（Excel/数据库）查阅所有在测表型的功耗参数，以便在测试前快速判断设备匹配度 |
| **Acceptance Criteria** | ① 列出所有在测表型（≥10 种）② 每款表型包含：型号、额定电压、额定电流、视在功率（VA）、备注 ③ 数据来源标注（规格书/实测）④ 缺失数据标注为"待补充"并指定获取负责人 |
| **Priority** | P2 |
| **Estimate** | 1.5 天 |
| **依赖** | 各型号规格书、研发确认 |

### Story 3.2: 设备带载能力匹配矩阵

| 字段 | 内容 |
|---|---|
| **User Story** | 作为测试部负责人/采购，我希望获得一个可视化的匹配矩阵，以便快速判断"哪些设备能测试哪些表型" |
| **Acceptance Criteria** | ① 矩阵 X 轴 = 在测表型（按功耗升序排列），Y 轴 = 可用设备（按带载能力升序排列）② 每个交叉点标注为绿色（安全：设备能力 ≥ 表型功耗 × 1.5）、黄色（边界：1.0-1.5 倍）、红色（不匹配：＜ 1.0 倍）③ 红色交叉点必须指定替代设备或升级方案 |
| **Priority** | P2 |
| **Estimate** | 1 天 |
| **依赖** | Story 3.1 功耗数据 |

### Story 3.3: 新设备采购验收 checklist 更新

| 字段 | 内容 |
|---|---|
| **User Story** | 作为设备采购，我希望在采购新测试设备时有一个更新后的验收 checklist，包含带载能力验证项，以避免采购到类似不达标设备 |
| **Acceptance Criteria** | ① Checklist 新增"带载能力测试"章节 ② 包含：测试方法（使用 2-3 种不同功耗的参考表）、通过标准（三相电流偏差 ≤ 0.5A）、测试记录模板 ③ 经测试部和采购部共同签字确认 |
| **Priority** | P2 |
| **Estimate** | 0.5 天 |
| **依赖** | Story 3.2 匹配矩阵（作为通过标准的参考） |

## Epic 4: 下一代产品可测试性优化

**Epic Hypothesis:** 我们相信通过在新产品设计评审阶段引入可测试性（DFT）检查，可以从源头上减少"产品设计导致测试困难"的问题——这类问题在量产后再修改的成本是设计阶段的 10 倍。

**拆分模式：** Defer Performance（先梳理 checklist，后续深入推动设计变更）

### Story 4.1: 硬件 DFT 评审 checklist 增加项梳理

| 字段 | 内容 |
|---|---|
| **User Story** | 作为测试工程师，我希望将 TU36 事件中暴露的可测试性问题（A-OUT 接口不便、功耗未标称等）整理为 checklist，以便在新产品设计评审中系统性地检查 |
| **Acceptance Criteria** | ① Checklist 列出 ≥5 个可测试性检查项 ② 每项包含：检查内容、为什么重要（附 TU36 案例作为教训）、检查方法 ③ 与研发对齐并确认可行性 |
| **Priority** | P3 |
| **Estimate** | 1 天 |
| **依赖** | 研发团队配合 review |

### Story 4.2: 与研发对齐 A-OUT 测试接口标准化方案

| 字段 | 内容 |
|---|---|
| **User Story** | 作为测试部，我希望在新产品的电路板设计阶段就和研发讨论测试接口的标准化——能否避免"需要断开焊接才能正常测试"的情况 |
| **Acceptance Criteria** | ① 与研发确认 A-OUT 走线设计的约束条件（"与电表功能设计相关，难以更改"的具体原因）② 探索折中方案：是否可以在 PCB 上预留测试点（Test Point）以减少焊接操作 ③ 形成一份"测试接口设计建议"文档 |
| **Priority** | P3 |
| **Estimate** | 1 天 |
| **依赖** | 研发潘工提供 A-OUT 设计约束说明 |

### Story 4.3: 新产品设计评审流程中嵌入可测试性节点

| 字段 | 内容 |
|---|---|
| **User Story** | 作为测试部负责人，我希望在设计评审流程中有一个正式的可测试性评审节点——测试部的意见在产品设计阶段就能被听到，而不是等到样机出来后才反馈 |
| **Acceptance Criteria** | ① 在现有设计评审流程中新增"可测试性评审"节点 ② 测试部被正式列为该节点的参与方 ③ 首个新产品评审中试点执行，收集反馈并迭代 |
| **Priority** | P3 |
| **Estimate** | 流程推动，持续 |
| **依赖** | 管理层/总工对评审流程调整的批准 |

## 对面试的价值

这份 WBS→Epic→Story 拆解证明你具备三项 PM 核心能力：

1. **结构化分解：** 将一个模糊问题（"设备不行"）分解为 4 个 Epic × 12 个 Story，每个 Story 有明确的 ACC、优先级、估算和依赖
2. **优先级判断：** P0（立即止血）→ P1（本周内去风险）→ P2（1-3 月制度化）→ P3（下一代设计前移）——展示了短期-中期-长期的节奏感
3. **Jira 实践：** 理解了 Epic→Story→Sub-task 的层级关系，可以直接上手 Jira
